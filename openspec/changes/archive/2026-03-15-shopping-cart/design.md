## 上下文

### 背景
- 当前系统仅支持单商品下单，用户需要反复选择商品、填写订单
- 订单与商品是一对一关系，一个订单只能包含一个商品
- 订单状态仅区分已处理和待处理，无法取消订单
- 购物车功能缺失，用户体验不佳

### 约束
- 购物车存储在前端 LocalStorage，无需后端持久化
- 用户必须登录才能使用购物车和下单功能
- 保持向后兼容：单商品直接下单功能需保留

## 目标 / 非目标

**目标：**
- 实现前端购物车功能（LocalStorage 存储）
- 支持多商品订单（一个订单包含多个商品项）
- 实现未处理订单取消功能
- 保持与现有下单流程的兼容性

**非目标：**
- 后端购物车数据同步（暂不考虑多设备同步）
- 库存锁定功能
- 订单超时自动取消
- 购物车数据持久化到数据库

## 决策

### 1. 购物车存储方案
**决策**：前端 LocalStorage

**理由**：
- 实现简单，无需后端改动
- 用户无需登录即可查看购物车
- 主流移动端电商常用方案
- 缺点：换设备无法同步（可接受，暂不需要）

### 2. 数据模型方案
**决策**：新增 OrderItem 关联表

**理由**：
- 结构清晰，订单与商品解耦
- 便于后续扩展（如订单项优惠、拆分发货等）
- 比 JSON 存储更具可查询性

### 3. 订单状态设计
**决策**：status 枚举调整

| 值 | 含义 |
|----|------|
| 0 | 待处理 |
| 1 | 已处理 |
| -1 | 已取消 |

**理由**：使用负数表示取消状态，与已处理状态区分开

### 4. 批量下单 API 设计
**决策**：新增 `/orders/batch` 接口

**理由**：
- 保持现有 `/orders` 单商品下单接口兼容
- 语义清晰，与购物车功能对应

## 风险 / 权衡

### 风险：LocalStorage 容量限制
→ 缓解措施：单个商品项数据量小，通常不会超出限制

### 风险：未登录用户添加购物车后登录，购物车丢失
→ 缓解措施：考虑后续版本支持后端购物车同步

### 权衡：订单取消是否需要管理员审核
→ 当前设计：用户可自行取消，无需审核（简单直接）

## 开放问题

1. 是否需要支持购物车数据同步到后端（多设备登录）？
2. 订单取消后是否需要发送通知（邮件/短信）？
